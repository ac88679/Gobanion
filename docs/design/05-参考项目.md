## 一、最值得深入研究的项目

下面几个项目与 Gobanion 的核心理念（DAG 驱动、目标→图、协调器+执行器分离）最接近，建议优先读源码。

### 1. open-multi-agent ★★★★★

- **仓库**: [open-multi-agent/open-multi-agent](https://github.com/open-multi-agent/open-multi-agent)（2026年5月，6k+ stars）
- **定位**: TypeScript 后端的多 Agent 编排框架
- **核心理念（与 Gobanion 最接近）**:
  - **"From a goal to a task DAG, automatically"** — 主 Agent 将目标自动分解为 DAG，与 Gobanion 的分层规划理念一致
  - Coordinator Agent 分解目标 → 自动并行化独立的子任务 → 综合结果
  - 对比 LangGraph JS：LangGraph 需要你声明式定义图，open-multi-agent 是在运行时由协调器自动分解
  - 三条运行时依赖，Node.js 后端即插即用
- **需要深入研究的设计**:
  - Coordinator 的分解算法怎么实现的（LLM 直接调用还是结构化 prompt？）
  - 怎么决定哪些任务可以并行、哪些必须串行
  - 任务回填机制：节点完成后结果如何综合
  - MemoryStore 设计（默认 in-process KV，可换 Redis/Postgres/Engram）
- **差异**: Worker 不是"自助认领"模式，是 Coordinator 分派好的

### 2. Augment Intent ★★★★

- **官网**: [augmentcode.com/intent](https://www.augmentcode.com/blog/intent-a-workspace-for-agent-orchestration)（2026年商业产品）
- **定位**: 多 Agent 代码编写的工作空间
- **核心理念**: Spec-Driven Development + 多 Agent 编排
  - Coordinator Agent → Specialist Agents → Verifier Agent
  - **"Living Spec"** — 随着实现进度实时更新的"活的需求文档"
  - Spec 双向更新：需求变了 → 所有活跃 Agent 收到更新 | Agent 完成 → Spec 自动更新
  - 每个 workspace 独立 git worktree 隔离
  - BYOA：支持 Claude Code / Codex / OpenCode
- **对 Gobanion 的启发**:
  - "Living Spec" 对应 Gobanion 的 DAG，都是共享的活的项目视图
  - Coordinator + Specialist + Verifier 的角色分工
  - 提交审核前的验证阶段（Verifier 检查是否满足 Spec）
- **局限**: 商业产品，不开源。macOS only

### 3. OpenAI Symphony ★★★★★

- **文章**: [OpenAI 官方博文 — An open-source spec for Codex orchestration](https://openai.com/index/open-source-codex-orchestration-symphony/)（2026年）
- **定位**: Codex 多 Agent 编排的开放规范
- **核心理念（与 Gobanion 高度吻合）**:
  - **"Agents only start working on tasks that aren't blocked"** — 跟 Gobanion 的子 Agent 认领 ready 节点完全一致
  - **基于 ticket 状态的状态机** — Linear 作为 DAG，任务状态流转决定执行顺序
  - **"If an agent crashes, restart it. If new work appears, pick it up"** — 自愈 + 动态吸纳
  - 围绕 **deliverables（issues/tasks/tickets）** 组织工作流，而不是 coding sessions
  - DAG 中的任务自然并行展开
- **对 Gobanion 的启发**:
  - 用项目管理系统（如 Linear/Jira）的 ticket 作为 DAG 节点的具体实现——可能比自建 DAG 更快起步
  - 状态机驱动、Agent 自检 ready 任务、崩溃重启的容错机制
  - 耦合 Linear 但设计理念非常接近
- **局限**: 规范层面，非可运行代码。强依赖 Linear（外部系统）

### 4. AWS Strands Agents SDK ★★★★

- **仓库**: [strands-agents/sdk-python](https://github.com/strands-agents/sdk-python)（2025年，Apache-2.0）
- **定位**: AWS 推出的开源多 Agent SDK（Amazon Nova + 外部模型）
- **核心理念**:
  - 定义了四种协作模式：Agent as Tools / Swarm / Agent Graph / Agent Workflow（DAG）
  - Agent Graph 模式下，每个节点是一个 Agent，边是通信/交接通道（单向或双向）
  - Agent Workflow 模式等同于 DAG of tasks，有显式依赖和执行顺序
  - 模型驱动（model-driven），几行代码定义一个 Agent
- **对 Gobanion 的启发**:
  - Agent Graph + Agent Workflow 的组合设计模式
  - DAG 节点 = Agent，边 = 通信通道 —— 与 Gobanion 的"DAG 节点 = 任务，边 = 依赖/协作"接近
  - 生产部署参考（Lambda / Fargate / EKS / Docker）
- **局限**: Agent 定义偏轻量，不强调"子 Agent 自助认领"和"DAG 作为共享工作台"

### 5. Microsoft Conductor ★★★★

- **文章**: [Microsoft Open Source Blog](https://opensource.microsoft.com/blog/2026/05/14/conductor-deterministic-orchestration-for-multi-agent-ai-workflows/)（2026年5月）
- **定位**: 确定性的多 Agent AI 工作流编排
- **核心理念**:
  - YAML 定义工作流 DAG，交互式 DAG 可视化（实时执行流动画）
  - 每个节点可点击查看 Agent prompt、模型、token 用量、成本、活动流和输出
  - "Your prompt is the code" → 用 prompt 而不是代码定义行为
  - 动态编排 vs 确定性编排的思考："if your task needs to restructure itself, let the LLM decide, but define the workflow skeleton"
- **对 Gobanion 的启发**:
  - YAML DAG 定义格式可作为参考
  - 可观测性设计：节点级活动流 + 成本显示
  - 灵活性 vs 确定性的平衡
- **局限**: 确定性的（不是运行时 LLM 动态规划 DAG）

### 6. Bernstein / Claude Squad ★★★

- **Augment Code 的综述**: [9 Open-Source Agent Orchestrators for AI Coding (2026)](https://www.augmentcode.com/tools/open-source-agent-orchestrators)
- **Bernstein**: 确定性调度器 + Janitor（lint/type check/test before merge）
- **Claude Squad**: tmux TUI，并行 Agent 会话管理，AGPL-3.0
- **对 Gobanion 的启发**:
  - 工作树（git worktree）作为隔离手段：每个 Agent 独立工作目录 + 独立的 branch + 共享 .git
  - 并行 Agent 的端口冲突问题（$EMDASH_PORT 方案）
  - 无人做完全自动化的合并冲突解析，所有工具都是"发现冲突→扔回给 Agent 重试"

## 二、分层记忆相关项目

### 7. Mem0 ★★★

- **仓库**: [mem0ai/mem0](https://github.com/mem0ai/mem0)
- **定位**: 智能记忆层，为 AI 应用提供持久化记忆
- **共鸣**:
  - 多层记忆抽象（用户/会话/Agent 维度）
  - embedding + 向量检索
- **局限**:
  - 没有 DAG 节点/任务维度的记忆索引
  - 不支持"每个 DAG 节点挂自己的执行历史"
  - 没有记忆生命周期（L1→L2 沉淀）的概念

### 8. LEGOMem ★★★★

- **论文**: [arXiv 2510.04851](https://arxiv.org/abs/2510.04851)
- **共鸣**:
  - Orchestrator memory + task agent memory 两层记忆分配
  - 卡住时可借助记忆重规划
  - 实验：带记忆的 Agent 减少 16.2% 执行步骤
- **局限**:
  - 只做了两层，没有 L3 元记忆
  - 没有 DAG 节点维度的索引

## 三、通信设施相关项目

### 9. NATS ★★★

- **仓库**: [nats-io/nats-server](https://github.com/nats-io/nats-server)
- **适合**: 节点间信号 + 全局广播的轻量基础设施
- **不适合**: 节点内频道（需要对话管理）

### 10. Matrix Protocol ★★★

- **适合**: 节点内频道的模型参考（每个 DAG 节点一个 Matrix 房间）
- **太重**: 起步不需要

## 四、工作流引擎参考（非 AI）

### 11. Temporal ★★★

- **定位**: 可靠的工作流编排引擎
- **共鸣**: DAG 执行过程中的动态增删步骤 + 重试 + 状态持久化
- **局限**: 工作流代码预定义，Agent 无自主性

### 12. Celery ★★★

- **定位**: 分布式任务队列
- **共鸣**: Worker 自领取任务 + 任务队列 + 结果后端
- **局限**: 无 DAG 依赖关系，Worker 无认知能力

## 六、Gobanion 的独特定位

从搜索到的所有项目来看，Gobanion 在以下组合上仍没有直接对标：

1. **DAG 作为共享项目全息图** — 所有 Agent 在同一 DAG 上读写，任务不是"分发"的而是"自取"的
2. **认领式执行** — 子 Agent 发现 ready 节点 → 自己判断是否匹配 → 认领执行
3. **三层通信设施** — 节点内频道 / 节点间信号 / 全局广播，按作用域分层
4. **DAG 节点索引记忆** — 所有过程记录、产出物、频道讨论挂在 DAG 节点上
5. **成本工程化** — 大主 + 小子，子 Agent 用小模型

**最接近的参考**: open-multi-agent（目标→DAG）、OpenAI Symphony（状态机 + 非阻塞执行）、Intent（Living Spec + Coordinator/VErifier）
