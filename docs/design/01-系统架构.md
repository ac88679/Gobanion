# 01 - 系统架构

> Gobanion 多 Agent 协作系统概要设计
> 版本: v0.6 | 日期: 2026-06-08

---

## 一、核心理念

**指挥官 + 专业化小队**：一个主 Agent（大参数量模型）负责总体的任务规划、指派、监控、评估和重规划，多个子 Agent（小参数量开源模型）通过 Skill 组装执行具体任务。

### 核心原则

- **主 Agent 负责智慧**：规划、决策、评估、重规划 — 需要强大推理能力。主 Agent 决定"做什么"和"谁来做"。
- **子 Agent 负责执行**：专注领域任务，Skill 可插拔组装，轻量高效。子 Agent 决定"怎么做"。
- **成本工程化**：主 Agent 单次大模型调用 ≈ 80-90% 成本，子 Agent 数量可弹性伸缩。
- **上下文隔离**：子 Agent 只看到自己任务相关的上下文，避免对话膨胀。
- **主 Agent 只关注结果**：不看子 Agent 执行细节，通过产出物和节点状态评估进展。出问题时通过 L1 失败记录了解原因。

## 三、DAG（项目全息图）

DAG 是整个项目的**活的共享工作台**。主 Agent 负责规划 DAG 结构（目标层 → 策略层 → 执行层），并将执行层节点指派给合适的子 Agent。

### 3.1 DAG 节点结构

每个节点就是一个具体任务：

```json
{
  "node_id": "T-003",
  "layer": "执行层",
  "goal": "实现用户认证模块",
  "assigned_roles": ["backend"],
  "required_skills": ["code_generator", "git_operator", "test_runner"],
  "dependencies": ["T-001", "T-002"],
  "collaborators": [],
  "acceptance_criteria": "- 所有测试通过\n- 代码评审无阻塞性缺陷",
  "status": "pending",
  "outputs": [],
  "assigned_agents": {},
  "channel_id": null
}
```

- `assigned_roles`: 主 Agent 规划时指定的角色需求数组（如 `["backend"]` 或 `["backend", "frontend"]`）
- `assigned_agents`: Dispatcher 分配具体的子 Agent 后填入 `{role: agent_id}` 格式
- `channel_id`: 当 `assigned_roles.length > 1` 时自动创建频道
- `collaborators` 保留，语义为跨节点协作关系

**节点状态流转**（MVP 跳过 reviewing，done → completed 直接终态）：

```
pending → ready → assigned → running → done → completed
                                          → failed → (重规划)
                               → stuck → (重规划)
                               → aborting → interrupted → (重规划)
```

| 状态 | 含义 | 谁可更改 |
|------|------|---------|
| `pending` | 依赖未满足，不可开始 | 主 Agent |
| `ready` | 所有依赖已满足，可指派 | 主 Agent (自动) |
| `assigned` | 已指派给某子 Agent，等待启动（超时 30s 未变 `running` → `failed`） | Dispatcher |
| `running` | 子 Agent 正在执行（每 10s 心跳，超时未收到 → `interrupted`） | 子 Agent |
| `done` | 子 Agent 自判完成（MVP 直接终态） | 子 Agent |
| `completed` | 复核通过，最终态（MVP: done 后直接 completed） | 主 Agent |
| `failed` | 执行出错 | 子 Agent (报错) |
| `stuck` | 频道内长时间无法对齐（频道未实现） | 子 Agent (上报)/主 Agent (检测) |
| `aborting` | 主 Agent 要求中止 | 主 Agent |
| `interrupted` | 已中止，记录中间产出 | 子 Agent |

### 3.2 分层规划（2+1 层）

主 Agent 一次 LLM 调用产出完整 DAG，不逐层与用户交互确认。

| 层级 | 产出 | 细化程度 | 决策者 |
|------|------|---------|--------|
| **目标层** | 用户输入的原始目标 | 顶层描述 | 用户 → 主 Agent |
| **执行层** | 可执行的 DAG 节点 | 每个节点的目标/角色/依赖/验收标准 | 主 Agent（一次产出）|
| **操作层** | Skill 级指令 | 子 Agent 内部的具体步骤 | 子 Agent 自行决定 |

**节点粒度经验规则**：一个节点应大致等于一个子 Agent 一次 LLM 会话能完成的工作量。超过则应考虑拆分为更细粒度的节点。

### 3.3 指派机制（Dispatch）

主 Agent 规划执行层时，为每个节点指定 `assigned_roles`（角色需求数组）。这不是指派到具体进程，而是标明"这里需要什么角色来干"。

实际的容器/进程匹配由 **Dispatcher（调度器）** 负责：

```
主 Agent 规划节点 T-003
  │  assigned_roles: ["backend"]
  │  required_skills: ["code_generator", "git_operator"]
  ▼
Dispatcher（事件驱动，异步）:
  1. 监听节点状态变为 ready
  2. 读取节点需求：roles=["backend"], skills=[...]
  3. MVP 阶段：直接 subprocess，不预热池
  4. 注入环境变量: NODE_ID, DAG_ID, TASK_DEF, ROLES, SKILLS, CHANNEL_ID...
  5. 异步启动 agent_runner.py（不等待就绪即返回）
```

这样主 Agent 的规划不与具体子 Agent 进程绑定，调度由 Dispatcher 在最后一刻完成。指派是异步的，Dispatcher 发出指令后立即返回处理下一个事件。

### 3.4 重规划

#### 触发条件

1. 某节点验收失败（failed）→ 主 Agent 分析 L1 失败记录，调整节点定义或重新指派
2. 执行结果与预期偏差大 → 拆/合/改节点
3. 子 Agent 卡住 / 超时（stuck）→ 调整角色或更换 Agent
4. 用户中途改需求 → 从目标层开始增量调整

#### 正在 running 的节点如何处理

规则：

- **变更不影响当前节点** → 继续跑，下次心跳同步即可
- **变更影响当前节点本身**（拆分/合并/删除/改角色）→ 主 Agent 发 `ABORT` 信号，节点状态变 `aborting`
- 子 Agent 收到 `ABORT` 后，**完成当前步骤即终止**，中间产出写入 L1 标记为 `interrupted`
- 不做强制 kill，给子 Agent 优雅中止窗口
- 主 Agent 等待节点变为 `interrupted` 后，再修改 DAG 重新指派

## 四、通信设施

DAG 是项目的"静态结构"，通信设施是项目的"动态神经"。两者互相配合，但不耦合：DAG 存状态，通信设施跑消息。

> **MVP 状态**：通信设施尚未实现。以下为设计文档，待 Phase 2 实施。

### 4.1 三种通信类型

| 类型 | 范围 | 用途 | 数据量 | 延迟要求 |
|------|------|------|--------|---------|
| **节点内频道** | 单个 DAG 节点内 | 同任务的子 Agent 间协作对齐（仅多 Agent 节点） | 结构化消息 | 低 |
| **节点间信号** | DAG 上下游 | 状态变更通知（"你前置完成了"） | 极小（状态 + 节点ID） | 高 |
| **全局广播** | 整个项目 | DAG 结构变更、需求变更 | 小（变更类型 + 影响范围）| 中 |

### 4.2 节点内频道（Task Channel）

频道**不是**节点的必备品。只有 `assigned_roles.length > 1` 的节点（需要多个子 Agent 协作）才创建频道。单 Agent 节点没有频道。

频道是**结构化协议通道**，不是自由讨论区。子 Agent 之间通过预定义的消息类型对齐。

#### 频道消息类型（通用领域抽象）

| 类型 | 含义 | 发送者 | 是否需要 response |
|------|------|--------|-----------------|
| `proposal` | 提出方案/协议/结论/接口定义 | 任意子 Agent | 是 |
| `response` | 回应：`accept` / `reject` / `counter-proposal` | 任意子 Agent | 否 |
| `issue` | 提出现待解决的问题 | 任意子 Agent | 否（需处理但有弹性）|
| `synced` | 双方确认当前对齐完成 | 任意子 Agent | 否 |
| `arbitration` | 主 Agent 仲裁结果 | 主 Agent | 否（终局消息）|

#### 核心规则

- **proposal 永远需要 response**。没有 response 的 proposal 不会有下一步，不会被归档。
- **归档条件**：某 proposal 收到了 `accept` 的 response，即自动归档。或收到 `reject + counter-proposal`，旧 proposal 失效，新 proposal 开始。
- **升级条件**：同一 proposal 出现 **3 轮以上** `reject`/`counter-proposal` → 触发 `stuck` 状态，主 Agent 介入仲裁。
- **仲裁采用举证摘要模式**：不读频道完整历史。频道 stuck 后，主 Agent 要求各子 Agent 各自提交 `evidence` 消息（200 字以内总结己方方案和理由），主 Agent 仅读 `evidence` 做裁决。一方证据不足可要求补充举证（最多 1 轮）。裁决结果以 `arbitration` 消息写入频道。
- **`arbitration` 只能由主 Agent 发**，子 Agent 收到后按仲裁结果执行，讨论结束。
- **`issue` 不需要 response**，但收到 issue 的子 Agent 如果无法解决，应发起一个新的 `proposal` 来覆盖。

#### 子 Agent 拉取规则

每次拉取频道仅获取：
1. 最新归档摘要（已 `synced` 的内容）
2. 所有未被归档的活跃消息

不拉取：已被归档覆盖的提案链、已被 `synced` 标记覆盖的历史。

#### 频道状态机

```
                    ┌────────────────────────────┐
                    │         active              │
                    │  (频道创建，正常讨论中)       │
                    └──────────┬─────────────────┘
                               │
                    ┌──────────┴──────────┐
                    │                     │
                    ▼                     ▼
            ┌──────────────┐   ┌──────────────────┐
            │   synced     │   │     stuck        │
            │ (双方对齐完成) │   │ (僵持/无法对齐)   │
            └──────┬───────┘   └────────┬─────────┘
                   │                    │
                   │            ┌───────▼────────┐
                   │            │  arbitration    │
                   │            │ (主Agent仲裁中)  │
                   │            └───────┬─────────┘
                   │                    │
                   ▼                    ▼
            ┌──────────────┐   ┌──────────────────┐
            │   archived   │   │   restructured   │
            │ (归档，频道关闭)│   │ (仲裁后重置/重建) │
            └──────────────┘   └──────────────────┘
```

- 频道生命周期跟随节点：节点创建且有多 Agent 协作时生成，节点完成后归档。
- 同一 DAG 节点的频道内出现 `stuck` → 节点状态变为 `stuck` → 主 Agent 介入。

### 4.3 节点间信号（DAG Signal）

轻量级的状态变更通知。当某节点状态变化时，向所有关心该节点的人（上下游节点、主 Agent、Dispatcher）发送信号：

```
信号: { type: "node.status_changed",
        node_id: "T-002",
        from: "running",
        to: "done",
        outputs: ["db/schema.sql"] }
```

信号的接收者自行决定如何响应：
- 下游节点检查自己是否满足 ready 条件
- 主 Agent 判断是否需要介入（如复核 reviewing 状态）
- Dispatcher 判断是否需要释放容器或重新调度
- 日志系统记录事件

### 4.4 全局广播（Global Event）

DAG 结构级别的变更：

```
广播: { type: "dag.structure_changed",
        change: "split",
        old_node: "T-003",
        new_nodes: ["T-003-A", "T-003-B"],
        affected: ["T-004", "T-005"] }
```

全局广播的消费者：
- 所有正在运行的子 Agent（判断自己是否被影响，受影响则中止或等待新指派）
- 持久化服务（记录 DAG 变更历史）
- 外部监控 / Web UI

### 4.5 技术选型建议

| 通信类型 | 起步方案 | 生产方案 |
|---------|---------|---------|
| 节点内频道 | Python asyncio.Queue / dict + 结构化消息 | NATS 或 Redis Pub/Sub |
| 节点间信号 | 回调函数 + 事件循环 | NATS（轻量消息） |
| 全局广播 | 同一事件循环内通知 | NATS / Redis Streams |

起步阶段全部走内存，不需要外部依赖。

## 五、子 Agent 设计

### 5.1 Skill 组装模型

```
子 Agent A (角色: 代码工程师)
┌─────────────────────────┐
│ LLM: Qwen3.5-9B         │
│                         │
│ Skill 组合:              │
│ ├─ code_generator       │
│ ├─ code_reviewer        │
│ ├─ git_operator         │
│ ├─ test_runner          │
│ └─ file_io              │
│                         │
│ 角色 Prompt: "你是资深   │
│ 后端工程师，擅长..."      │
└─────────────────────────┘
```

子 Agent 启动时从环境变量获取：`NODE_ID`、`DAG_ID`、`ASSIGNED_ROLES`、`REQUIRED_SKILLS`、`TASK_DEFINITION_JSON`、`CHANNEL_ID`（如有）、`MASTER_API`、`LLM_ENDPOINT`、`LLM_MODEL`，然后立即开始执行。子 Agent 通过 localhost HTTP API 与主 Agent 通信。

### 5.2 子 Agent 工作流程（Loop Agent）

```
1. Dispatcher 指派节点到当前容器，注入环境变量
2. agent_runner.py 启动
3. 读取节点定义（goal、acceptance_criteria、assigned_roles、skills）
4. 将节点状态设为 running（通过 HTTP API）
5. 如果 assigned_roles.length > 1 → 加入频道（MVP 未实现频道通信）
6. 一次性规划全部执行步骤（LLM tool calling，fallback 文本解析）
7. 逐步骤执行（Loop Agent 闭环）:
   ├─ 执行 → ok → 收集产出物，继续下一步
   └─ 执行 → fail → LLM 分析错误，决策恢复策略:
        ├─ retry          → 重试当前步
        ├─ fix_and_retry  → 插入修复步，再重试
        ├─ skip           → 跳过当前步
        └─ abort          → 标记节点 failed
       ↑ 每步最多重试 3 次，全局最多 20 步
       ↑ 修复步失败 → 直接 abort（不递归恢复）
8. 每执行一步 → 写入 L1（JSONL 文件，位于 _workspace/{dag_id}/{node_id}/_memory.jsonl）
9. 按验收标准逐条自检，生成自检报告
10. 完成 → 产出物发布到 _outputs/{dag_id}/ 目录（下游节点可访问）
11. 标记节点状态为 done（MVP 跳过 reviewing，直接终态）
12. 退出
```

### 5.3 子 Agent 生命周期

```
节点被指派 → agent_runner 启动 → 执行中
  │
  ├── 正常完成 → done → completed（MVP 跳过 reviewing）→ 退出
  │
  ├── 执行出错 → Loop Agent 恢复闭环:
  │     ├─ retry          → 重试，最多 3 次 → 成功则继续，否则
  │     ├─ fix_and_retry  → 插修复步后重试 → 同上
  │     ├─ skip           → 跳过，继续后续步骤
  │     └─ abort          → failed → 写入 L1 失败记录 → 退出
  │
  ├── 收到 ABORT → 完成当前 step → interrupted → 退出
  │
  └── 频道僵持（MVP 未实现）
```

子 Agent 的生命周期不大于单个节点的生命周期。一个子 Agent 完成一个节点后进程即退出。主 Agent 不会等子 Agent，指派后继续处理其他事务。

### 5.4 小模型选型参考

| 模型 | 参数量 | 适合场景 | 经济性 |
|------|--------|---------|-------|
| Qwen3.5-9B | 9B | 代码生成/通用 | 极好 |
| Qwen2.5-7B | 7B | 通用文本 | 极好 |
| Llama-3.2-3B | 3B | 简单分类/格式化 | 免费级 |
| DeepSeek-Coder-V2-Lite | 16B | 复杂代码 | 好 |
| Phi-3.5-mini | 3.8B | 设备端推理 | 免费级 |

### 5.5 LLM 调用失败回退

| 失败类型 | 重试次数 | 策略 |
|---------|---------|------|
| 格式错误（非 JSON） | 2 次 | 注入原始输出，要求"修正格式" |
| 内容不符合约束 | 1 次 | 注入更严格的约束说明 |
| 连续失败 / 超时 | 降级 | 用预设模板 / 报"无法完成"给主 Agent |

可配置 `FALLBACK_MODELS` 列表，主模型失败 3 次后自动切换下游模型。

### 5.6 日志实时流

子 Agent 的每条日志通过 `print(flush=True)` 实时输出到 stdout，Dispatcher 通过 `subprocess.PIPE` + 后台 reader 线程逐行转发到服务器日志。同时写入 `_logs/{dag_id}/{node_id}.log` 文件持久化。

**日志 API**：`GET /api/v1/dag/{dag_id}/nodes/{node_id}/logs?tail=N&offset=N`

前端可通过轮询该接口实时查看子 Agent 执行进度。

### 5.7 Session 上下文组装

子 Agent 每步执行前读取 `_workspace/{dag_id}/{node_id}/_memory.jsonl`，构建累积步骤历史：

```
已完成步骤：
  步骤 1: write_file → ok  - Wrote routes.py (5107 bytes)
  步骤 2: execute_code → fail  - exit=1, output=...
```

各 prompt 模板（Jinja2 渲染）均注入 `{history}` 变量，确保 LLM 每一步都知道之前做了什么。

Prompt 全部外置到 `backend/prompts/` 目录下的独立 `.md` 文件，用 Jinja2 模板引擎渲染：

| Prompt 文件 | 用途 |
|------------|------|
| `agent_step_planner.md` | 子 Agent 步骤规划 |

### 5.8 执行安全

- **命令黑名单**：`setx /M`、`sudo`、`choco install` 等需要管理员权限的命令被硬拦截
- **输出过滤**：LLM 输出的 markdown 代码块包裹（```` ```python ````）写入文件前自动剥离
- **uv run 自动前缀**：检测到 uv 环境时，`python` 开头的命令自动加 `uv run` 前缀
- **端口安全**：禁止子 Agent 启动 HTTP 服务（主后端已占用 5000 端口）
- **禁止后台进程**：禁止 `&`、`nohup` 等后台运行方式
- **API 测试要求**：验证 HTTP 接口时使用框架测试客户端（如 `app.test_client()`）

### 5.9 环境检测

子 Agent 启动时自动检测运行环境，注入到 LLM 上下文：

```python
self._os_type = "Windows" / "Linux/Mac"
self._pip_cmd = "uv pip install" / "pip install" / ""
self._pip_run_prefix = "uv run" / ""
self._has_node / self._has_npm  # Node.js 环境检测
```

检测结果影响 planner、executor、recovery 三处 prompt。

## 七、关键流程

### 主 Agent 创建项目

```
1. 用户输入需求
2. 主 Agent 一次 LLM 调用产出目标层 + 执行层 DAG（每个节点指定 assigned_roles）
3. DAG 发布 → 全局广播通知（DAG 结构公开，子 Agent 可读）
4. Dispatcher 事件驱动，将 ready 节点异步指派给匹配的子 Agent
```

### 节点执行 & 完成

```
1. Dispatcher 将 ready 节点指派给空闲子 Agent → 状态: assigned
2. 子 Agent 启动 → 状态: running → 广播信号（通过 HTTP API）
3. 子 Agent 一次性规划执行步骤（LLM tool calling）→ 逐步骤执行 → 每步写 JSONL
4. 产出物发布到 _outputs/{dag_id}/ 目录
5. 子 Agent 按验收标准逐条自检，生成自检报告
6. 标记状态: done（MVP 跳过 reviewing，直接终态）
7. 下游节点检查依赖是否满足 ready
```

### 频道对齐流程（MVP 未实现）

```
节点 T-003 需要子Agent A + B 协作创建频道

子Agent A                    频道                   子Agent B
  │                          │                        │
  ├── proposal("接口: ...")──→│                        │
  │                          │── proposal("接口:...")→│
  │                          │                        ├── response("accept")
  │                          │←── response("accept")──│
  │  ← 自动归档 →            │                        │
  │                          │                        │
  ├── synced("对齐完成")─────→│                        │
  │                          │── synced("对齐完成")──→│
  │                          │                        │
  └──── 归档，各自继续执行 ──→┘                        │
```

### 频道冲突升级流程（MVP 未实现）

```
子Agent A: proposal("方案X")
子Agent B: response("reject + 方案Y")
子Agent A: response("reject + 方案X-改")
子Agent B: response("reject + 方案Y-改")
子Agent A: response("reject + 方案X-改-再改")
  │
  └── 3轮 reject 了！
  │
  ▼
节点状态 → stuck
  │
  ├── 主 Agent 要求双方提交举证摘要
  ├── 子Agent A 发 evidence（200字方案X+理由）
  ├── 子Agent B 发 evidence（200字方案Y+理由）
  ├── 主 Agent 仅读 evidence 做裁决
  │   └── 证据不足？→ 可要求补充举证（最多 1 轮）
  ├── 主 Agent 裁决 → 发 arbitration 消息
  ├── 双方按仲裁结果执行
  └── 节点状态 → running
```

### 重规划（变更正在 running 的节点）

```
1. 主 Agent 决定修改 T-003（拆分/合并/更换角色）
2. 检查 T-003 当前状态：
   ├── pending/ready：直接改
   ├── running/assigned：发 ABORT 信号，状态变 aborting
   └── stuck：记录仲裁理由后 abort
3. 子 Agent 收到 ABORT → 完成当前 step → 停止
4. 中间产出写入 L1，标记 interrupted
5. 节点状态变为 interrupted
6. 主 Agent 确认后修改 DAG（增/删/改节点）
7. 全局广播结构变更
8. Dispatcher 按新 DAG 继续指派
```

### 主 Agent 不依赖子 Agent 实时在线

主 Agent 进程重启后：
1. 从 SQLite 重建 DAG 状态
2. 检查所有状态为 `running` / `assigned` 的节点
3. 对应的子 Agent 进程是否存活？
   ├── 已死 → 标记为 `interrupted`，重新指派
   └── 仍在运行 → 保持状态不变
