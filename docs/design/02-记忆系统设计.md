# 02 - 记忆系统设计

> 多 Agent 系统的记忆分层、存储、回滚与生命周期管理
> 版本: v0.4 | 日期: 2026-06-08

---

## 一、记忆总览

记忆以 **DAG 节点为第一索引**，Agent 维度为第二索引。每个 DAG 节点的执行过程、产出物、通信记录都直接挂在该节点上。

| 层级 | 名称 | 生命周期 | 作用域 | 存储 |
|------|------|---------|--------|------|
| L0 | 瞬时记忆 | 从会话开始到结束 | 当前 Agent 的当前会话 | 内存 |
| L1 | 过程记忆 | 节点生命周期 | 每个 DAG 节点 + 节点内频道 | JSONL 文件（MVP）/ SQLite（规划） |
| L2 | 沉淀记忆 | 长期 | 全局，跨 Agent 可读 | 向量库 + 数据库 |
| L3 | 元记忆 | 长期 | 系统级 | 结构化数据库 |

---

## 二、四层记忆详解

### L0 - 瞬时记忆（工作上下文中）

```
适用范围: 从会话开始到结束
作用域:  当前 Agent 的当前会话
存储:    内存 (带 TTL)
内容:
├─ 当前任务的完整对话上下文
├─ 当前步骤正在执行的指令序列
├─ 已生成的中间产出
├─ 子 Agent 当前对话历史 (上下文窗口)
├─ 从 L1/L2 召回的相关片段
├─ 节点内频道实时讨论内容
├─ 临时中间数据 (文件路径、API 响应等)

特点:
- 不持久化 (崩溃即丢失)
- 可基于 L1 重建对话上下文
```

### L1 - 过程记忆（节点执行流水账）

以 **DAG 节点为单位**，每个节点挂自己的过程记忆。通信设施的节点内频道讨论也在这一层归档。

> **MVP 实现**：当前 L1 写入 `_workspace/{dag_id}/{node_id}/_memory.jsonl`，不经过 SQLite。子 Agent 的 `write_memory()` 方法直接追加 JSON 行到文件。`_build_step_history()` 从 JSONL 读取构建步骤历史。L1 HTTP API 端点待 Phase 2 实现。

```
适用范围: 节点生命周期
作用域:  per-DAG-Node (以 node_id 为主要键)
存储:    JSONL 文件（MVP）/ SQLite + 文件系统（规划）
内容:
├─ 节点执行日志 (分步骤)
│   ├─ 步骤1: [操作, 输入摘要, 输出摘要, 耗时, 状态]
│   ├─ 步骤2: [操作, ...]
│   └─ ...
├─ 节点内频道通信摘要
│   ├─ 协作 Agent 之间的讨论记录
│   ├─ 对齐的接口定义
│   └─ 关键决策点
├─ 失败记录 (失败原因、尝试次数、回退路径)
├─ 关键决策点及理由
├─ 中间产物 (代码片段、文档草稿、测试结果)
├─ 节点上下游通信记录摘要
├─ 时间戳 + 域标签
```

**表结构：**

```sql
TABLE node_memory (
  id            UUID PRIMARY KEY,
  dag_id        TEXT,
  node_id       TEXT,            -- DAG 节点 ID，主索引
  session_id    TEXT,
  step          INT,
  actor         TEXT,            -- 哪个子 Agent 执行的
  action        TEXT,            -- think/tool/code/result/communicate/self_check
  input_summary TEXT,
  output_summary TEXT,
  output_path   TEXT,
  status        TEXT,            -- success/fail/partial
  reason        TEXT,            -- 成功原因 / 失败原因
  self_check    TEXT,            -- 子 Agent 自检报告 JSON（有 done 标记时）
  token_cost    INT,
  elapsed_ms    INT,
  tags          TEXT[],
  embedding     VECTOR(?),
  created_at    TIMESTAMP
);

TABLE node_channel_memory (
  id            UUID PRIMARY KEY,
  dag_id        TEXT,
  node_id       TEXT,            -- 关联的 DAG 节点
  from_agent    TEXT,
  to_agent      TEXT,
  message_type  TEXT,            -- proposal/response/issue/synced/arbitration/evidence
  content_summary TEXT,
  content_path  TEXT,            -- 完整内容存文件
  decision      BOOLEAN,        -- 是否是关键决策
  created_at    TIMESTAMP
);

TABLE node_blockers (
  id            UUID PRIMARY KEY,
  dag_id        TEXT,
  node_id       TEXT,
  block_type    TEXT,            -- dependency/external/error/ambiguity
  description   TEXT,
  resolved_by   TEXT,
  resolved_at   TIMESTAMP
);
```

### L2 - 沉淀记忆（做完后的知识沉淀）

> **MVP 未实现**。以下为设计阶段规划。

节点完成后，其 L1 中的有价值信息可沉淀到 L2，形成跨节点可复用的知识。

```
适用范围: 长期 (智能汰换)
作用域:  全局 (跨 Agent 可读)
存储:    向量数据库 + 结构化数据库 + 文件系统
内容:
├─ 节点最终产出物 (方案、代码、文档)
├─ 节点总结 (做了什么、为什么这么做、关键决策、教训)
├─ 成功模式 (可复用的套路)
├─ 失败教训 (下次要避免)
├─ Agent 画像积累 (某个角色在项目中积累的领域知识)
├─ 跨节点关联 (T3 的输出被 T7 引用了)
├─ 领域最佳实践 (Agent 总结出来的)
├─ 节点间协作模式记录
```

**沉淀流程：**

```
节点完成 (done)
  │
主 Agent 审核通过后，根据 L1 和产出物生成节点总结
  │
存入 L2 存储 + 生成 embedding
  │
触发全局广播：新知识可用
```

> 子 Agent 退出前不做节点总结。自检报告随 `done` 信号写入 L1，主 Agent 在 `reviewing` 阶段决定是否需要 L2 沉淀。如需沉淀，主 Agent 自行创建总结或指派一个专门的总结节点去完成。

**表结构：**

```sql
TABLE knowledge_base (
  id            UUID PRIMARY KEY,
  node_id       TEXT,            -- 来源节点
  source_type   TEXT,            -- node/agent/reflection
  source_id     TEXT,
  title         TEXT,
  summary       TEXT,
  content       TEXT,
  content_path  TEXT,
  domain        TEXT,            -- code/design/analysis/...
  quality_score FLOAT,
  usage_count   INT,
  related_node_ids TEXT[],       -- 关联的其他节点
  embedding     VECTOR(1536),
  tags          TEXT[],
  created_at    TIMESTAMP,
  last_used_at  TIMESTAMP
);
```

### L3 - 元记忆（系统级洞察）

> **MVP 未实现**。以下为设计阶段规划。

```
适用范围: 长期
作用域:  系统级
存储:    结构化数据库
内容:
├─ Agent 画像模型
│   ├─ agent_A: [擅长领域, 成功率, 平均耗时, token 消耗]
│   ├─ agent_B: ...
├─ Skill 效果统计
│   ├─ code_generator: [成功率, 返修率, ...]
│   ├─ ...
├─ 任务模式库
│   ├─ "为 Python 项目写登录" → [最佳路径, Skill 组合, 预计耗时]
│   ├─ ...
├─ 协作冲突记录
│   ├─ "T3 和 T5 写了同名文件" → [原因, 预防措施]
│   ├─ ...
├─ 系统调优建议 (token 预算、并发数、缓存策略)
```

L3 的数据帮助主 Agent 做**角色推荐**：给一个新的 DAG 节点，主 Agent 查 L3 决定推荐哪个子 Agent、推荐哪些 Skill。

**表结构：**

```sql
TABLE agent_profile (
  agent_id        TEXT PRIMARY KEY,
  role            TEXT,
  base_model      TEXT,
  skill_ids       TEXT[],
  success_rate    FLOAT,
  avg_task_time   INT,           -- 秒
  avg_token_per_task INT,
  task_history    TEXT[],        -- 最近 N 个节点 ID
  best_at         TEXT,
  weak_at         TEXT,
  last_updated    TIMESTAMP
);
```

## 三、记忆召回机制

### 3.1 默认召回路径

```
                 用户/主 Agent 查询
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
   ┌─────────────┐ ┌──────────┐ ┌──────────┐
   │ DAG 节点查询 │ │向量检索  │ │结构查询  │
   │ node_id 直查 │ │(RAG)    │ │标签/域  │
   │ 所有关联记忆 │ │查 L2     │ │时间范围  │
   └─────────────┘ └──────────┘ └──────────┘
          │              │              │
          └──────────────┴──────────────┘
                         │
                    Top-K 结果
```

**以 DAG 节点为第一入口**：查某个节点时，直接拿到该节点的所有 L1 过程记录、L2 沉淀知识、通信记录。

### 3.2 场景驱动召回策略

| 场景 | 召回路径 | 数据来源 |
|------|---------|---------|
| 新节点开始 | 向量检索 | 相似节点的 L2 知识 |
| 卡住/失败 | 查找失败模式 | 同类型节点的 L1 过程记忆 |
| 主 Agent 重规划 | 全量查询 | L3 元记忆 + L2 知识 |
| 跨节点关联 | DAG 节点追踪 | 沿 DAG 上下游遍历 |
| 节点常规执行 | 非必要不召回 | 无 |

## 四、记忆生命周期全流程

```
主 Agent 创建 DAG 节点
  │
  ├─ 主 Agent 查 L3 选 Agent → 写入 recommended_role
  ├─ 查 L2 召回相关知识 → 注入子 Agent 初始上下文
  │
  ▼
子 Agent 执行节点
  │
  ├─ 每一步 → 写入 L1 (过程记忆) — MVP 直接写 JSONL 文件，规划 HTTP API
  ├─ 节点内频道讨论 → 写入 L1 channel_memory
  ├─ 子 Agent 不直接查 L1（不跨节点检索 L1）
  │
  ▼
节点完成 (done)
  │
  ├─ 子 Agent 生成自检报告（按验收标准逐条自检）→ 写入 L1
  ├─ 写入 L1 完成标记
  │
  ▼
主 Agent 审核（reviewing）
  │
  ├─ 通过 → 主 Agent 根据 L1 和产出物生成节点总结，沉淀到 L2
  ├─ 失败 → 标记 L1 失败，写 L2 教训，触发重规划
  │
  ▼
全局同步 (异步后台)
  ├─ 更新知识库 embedding
  ├─ 更新 L3 AgentProfile
  ├─ 更新 L3 模式库
  ├─ L1 归档 / 淘汰
  └─ 重建 embedding 索引
```

## 五、记忆压缩与淘汰

```
L1 过程记忆压缩:
├─ 知识蒸馏: 节点完成后 L1 关键信息沉淀到 L2
├─ 窗口截断: 子 Agent 每次只保留最近 N 步上下文
├─ 摘要化: 旧的中间步骤 → 自动摘要 (小模型完成)
└─ 自动淘汰: 超过 30 天的 L1 且未被引用 → 直接删除

L2 知识淘汰:
├─ 使用计数: 低于阈值的冷知识归档
└─ 质量评分: 低分知识标记为 "需验证"
```

## 六、设计原则

1. **写多读少**：L1 写频繁，每一步都写但不要求立即读到；L2 写少，但读频繁，要快。
2. **成本可感知**：时刻清楚"这个查询花掉了多少 token"，写入时预判读取开销。
3. **时间衰减加权**：所有记忆有时效，近期的记忆自动加权。
4. **范围最小化**：每个 Agent 只看到自己需要的记忆范围。主 Agent 看 L3，子 Agent 不直接读 L1 细节。
5. **来源可追溯**：所有沉淀知识都带"来源节点 ID"，保证可追溯。
