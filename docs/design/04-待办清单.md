# 04 - 待办清单 & 设计决策

> 设计阶段待决策问题
> 版本: v0.5 | 日期: 2026-06-01

---

## ✅ 已决策（已写入架构文档）

### 1. DAG 存储与并发访问

**决策**: SQLite（主 Agent 进程持有唯一写连接），子 Agent 通过 DAG Service API 访问，不直连 DB。

- 子 Agent 只改自己节点的状态（`running → done`），不改 DAG 结构
- SQLite 行级锁保障认领唯一性（指派模式下无需认领）

### 2. 子 Agent 获取任务：指派模式（非认领）

**决策**: 精细派单模式。主 Agent 指定 `assigned_roles[]`，Dispatcher 匹配空闲容器。

- 主 Agent 决定"做什么"和"谁来做"，子 Agent 决定"怎么做"
- 不存在抢单冲突，不需要待命监听

### 3. 节点内频道通信机制

**决策**: 中心化中转，主 Agent 进程内 ChannelService。

- 仅 `assigned_roles.length > 1` 的节点创建频道
- 消息持久化 = 同时写入 L1

### 4. 频道上下文膨胀控制

**决策**: 结构化消息协议（5 种类型）+ 自动归档 + stuck 升级机制（举证摘要仲裁）。

### 5. DAG 节点产出物物理存储

**决策**: 项目 `_outputs/` 目录，按 `{node_id}/{step}_{filename}` 组织。

### 6. 记忆存储服务

**决策**: MVP 内嵌主 Agent 进程（同一份 SQLite，不同表），Phase 2 独立。

### 7. 子 Agent is_done 自判可信度

**决策**: 增补 `reviewing` 状态，主 Agent 异步复核。

- 子 Agent 按验收标准逐条自检，生成自检报告随 `done` 信号提交
- 主 Agent 复核子 Agent 的自检报告（抽查，不重跑全部）
- 复核通过 → `completed`，不通过 → `failed`

### 8. 预热池角色映射

**决策**: 字符串标签（"backend"/"frontend"/"designer"/"analyst"/"generic"），精确匹配 → generic 兜底 → 创建。

### 9. Skill 版本与依赖管理

**决策**: MVP 跳过版本管理，加后缀命名约定。

### 10. 子 Agent LLM 调用失败处理

**决策**: 三级回退（格式错误→2 次，不满足约束→1 次，连续失败→降级报错）。

### 11. 重规划时正在 running 的节点

**决策**: ABORT 优雅中止，完成当前 step 后退出。

### 12. Git 集成

**决策**: 子 Agent 不直接操作 Git，主 Agent 在节点 completed 后统一 commit。

### 13. 用户交互接口

**决策**: Phase 1 纯 CLI + 日志，Phase 2 WebSocket，Phase 3 Web UI。

### 14. 多租户

**决策**: Phase 1 单项目，Phase 3 再扩展。

## ✅ 已解决：设计冲突

### C1. DAG 节点多角色支持

**决策**: `assigned_role` → `assigned_roles`（数组）。

- `collaborators` 保留，语义为跨节点协作
- 频道按 `assigned_roles.length > 1` 创建
- Dispatcher 为每个角色分配一个容器

### C2. 容器回池后上下文残留

**决策**: MVP 每节点新建 subprocess，不复用。Phase 2 Docker 预热池按角色分池。

### C3. 主 Agent 仲裁需读频道历史

**决策**: 仲裁用举证摘要模式。子 Agent 各自写 `evidence`（200 字以内），主 Agent 仅读举证做裁决。

### C4. 子 Agent 查其他节点 L1

**决策**: 子 Agent 不查 L1。查 L2（知识沉淀后 RAG），由主 Agent 在指派时注入。

### C5. MVP subprocess 写 L1 的实现路径

**决策**: 统一走 localhost HTTP API。主 Agent 启动 HTTP 服务，子 Agent 通过 localhost 调用。

## ✅ 已解决：设计不明确

### U1. 主 Agent 分层规划流程

**决策**: 2+1 层（目标层 → 执行层 → 操作层），去掉策略层。

- 主 Agent 一次 LLM 调用产出完整执行层 DAG
- 不逐层与用户交互确认
- 参考 open-multi-agent 的 goal → DAG 直出模式

### U2. 子 Agent 自主规划：一次性规划完

**决策**: 子 Agent 在开始时一次性规划完所有步骤，逐个执行。执行中如遇规划不合理可申请重规划（异常路径）。

### U3. done → reviewing 转交机制与审核标准

**决策**:
- 子 Agent 标记 `done` 时附带**自检报告**（按验收标准逐条 pass/fail + 证据）
- 主 Agent EventHandler 监听到 `done` 后自动改为 `reviewing`
- 主 Agent 复核子 Agent 的自检报告（抽查，不重跑全部）
- 复核通过 → `completed`，不通过 → `failed`，自检报告 + L1 失败记录触发重规划

**自检报告格式**:
```json
{
  "node_id": "T-003",
  "self_check": [
    {"criterion": "登录接口返回 token", "result": "pass", "evidence": "测试返回 200 + token"},
    {"criterion": "密码 bcrypt 加密", "result": "fail", "evidence": "使用 SHA256 而非 bcrypt"}
  ],
  "overall": "partial_fail",
  "summary": "核心功能完成，安全合规未通过"
}
```

### U4. Dispatcher 触发时机

**决策**: 事件驱动，异步指派。

- 节点状态变为 `ready` 时自动触发 Dispatcher
- Dispatcher 检查空闲容器池，匹配 `assigned_roles`，注入完整上下文后指派
- 指派是异步的，不等待容器就绪
- `ready` 时要保证子 Agent 拿到完整的指令和可用的运行环境

### U5. MVP 阶段 L1 表范围

**决策**: 三张表全部建（`node_memory`、`node_channel_memory`、`node_blockers`）。
- MVP 主 Agent HTTP API 端点全部就绪，子 Agent 按需写入

## 📋 实施计划（v0.5）

```
Phase 1 - 核心验证 (MVP)
├─ 主 Agent 分层规划（目标层 → 执行层一次性产出）
├─ DAG Service（SQLite，主 Agent 进程内）
│   ├─ Dispatcher 事件驱动异步指派（assigned_roles 数组匹配）
│   ├─ 节点状态机含 assigned / reviewing / aborting 状态
│   ├─ assigned 超时 30s 未变 running → failed
│   ├─ running 每 10s 心跳，超时未收到 → interrupted
│   └─ ABORT 优雅中止
├─ 子 Agent subprocess 运行
│   ├─ localhost HTTP API 与主 Agent 通信
│   ├─ 一次性规划所有执行步骤
│   ├─ Skill 按命名约定加载
│   ├─ LLM 三级失败回退
│   └─ 完成后提交自检报告（自检 + done 信号）
├─ 主 Agent 异步复核
│   ├─ 读取子 Agent 自检报告 → 抽查复核
│   └─ 通过 → completed / 不通过 → failed
├─ L1 过程记忆（三张表全建：node_memory / channel_memory / blockers）
├─ 节点内频道（结构化消息协议 5 种类型 + 举证摘要仲裁）
├─ 产出物管理（项目 _outputs/ 目录）
├─ 重规划引擎（DAG 增删改 + ABORT + stuck 仲裁）
├─ 最简单 Demo: 主 Agent 拆任务 → 子 Agent 接活 → 写文件 → 完成
│   （先不用 Docker、L2/L3 记忆、Web UI）

Phase 2 - 完善基础设施
├─ Docker 容器化子 Agent + 预热池（按角色分池）
├─ 混合运行模式（内存轻量 + Docker 重量）
├─ Memory Service 独立化（含 L2 向量检索）
├─ L2 沉淀记忆 + 主 Agent 审核沉淀流程
├─ Skill Registry + 版本解析器
├─ WebSocket 实时推送 DAG 状态
├─ Git 集成（主 Agent 统一 commit + push）

Phase 3 - 生产增强
├─ L3 元记忆（Agent 画像、模式库）
├─ AgentProfile 与预热池角色动态映射
├─ Web UI / 用户交互接口
├─ MCP 协议端点暴露
├─ 多租户 / 多项目
```
